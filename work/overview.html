REAP
Ruby Project Management

Introduction

Reap is an advanced life-cycle based build system and project mangement tool. Reap organizes the various tools Ruby developers need to manage their projects, from code generation to releasing and promotion.

Reap is the original project management application for Ruby developers. It was first developed in 2005 as a simple add-on to Rake. Since then it has gone through a number of transformations in an exploration for the best possible system. The final product, Reap 10, is culmination of this work.

NOTE: The current release is actually the 9.x series. This is a beta series before the final release of Reap 10.

Installation

Via RubyGems:

      $ gem install reap

With the source code:

      $ tar -xzf reap-x.y.z.tar.gz
      $ cd reap-x.y.z
      $ sudo setup.rb

Microsoft Windows users, you'll have to use ruby setup instead.

Usage

Reap is a life-cycle based build system. This means there is a linear set of actions that are taken with each execution of +reap+. These cycles are called *pipelines* --since they pass the project from one action to the next in order. Reap can support any variety of pipelines, but three pipelines are provided out of the box. The most central of which is the 'main' pipeline.

Each pipeline consits of a set (or sets) of ordered phases. For instance, the main pipeline consists of the following ordered phases:

      configure
      generate
      analyize
      compile
      validate
      document
      package
      release
      promote
      archive

To execute reap, one provides the pipeline name and the phase desired, separated by a colon (':').

      $ reap main:document

Since +main+ is Reap's primary pipeline, it is the default and can be omitted. The above can therefore be written more conveniently: 

      $ reap document

When the above command is run, Reap will begin with the top phase (configure) and work it's way down executing every phase in order until it completes the phase specified on the command line (in this case 'document').

Now you might wonder what the various phases actually do. That all depends on what services are available and which and how they are configured. Services are contected to pipelines and phases. For example, Reap provides a built-in RDoc service. This is contected to the 'main' pipeline's document phase, as well as to the 'site' pipeline's document phase. So if an RDoc service is defined in a project's reap configuration, the running 'read document' generated RDoc's for the projects, as will any phase after 'document'.

For more detailed usage documentation, please refer to the tutorial and RDocs.
